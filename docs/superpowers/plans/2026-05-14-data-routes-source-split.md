# Data Routes Source Split Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Move the curated `data` variant routes from `dist/data/routes.js` (hand-edited, lives in the build-output directory) into a typed TypeScript source at `src/data/routes.ts`, so that `dist/` becomes 100% reproducible from `src/`.

**Architecture:** The data pipeline currently treats `dist/data/routes.js` as both upstream input (read by `gen-data-types.ts`) and downstream output (consumed by package users via `@heroku/types/data/routes`). This refactor breaks the conflation: `src/data/routes.ts` becomes the typed source-of-truth, and the data CLI gains a step that transpiles it into `dist/data/routes.js` + emits `dist/data/routes.d.ts` via the TypeScript compiler API. The existing `types.d.ts` emission is unchanged. The end result is the invariant "anything in `dist/` is generated by `npm run generate*`" — `rm -rf dist/ && npm run generate && npm run generate:data` becomes a safe operation.

**Tech Stack:** TypeScript 6, Node 22, vitest, tsx, TypeScript compiler API (`typescript` package — already a devDependency).

---

## File Structure

**New files:**
- `src/data/routes.ts` — typed source for the curated data-variant route grouping. Imports `RouteDefinition` from `../gen/schema-types.js`. Each per-resource export uses `as const satisfies Record<string, RouteDefinition>` so the literal shape is preserved (consumers get string literal types) while the type system enforces correctness at authoring time.
- `src/gen/emit-typed-source.ts` — small helper that wraps `ts.createProgram`/`program.emit` to compile a single typed `.ts` file to `.js` + `.d.ts`. Single responsibility: typed source → emitted artifacts. Reusable if any future variant needs the same treatment.
- `src/gen/emit-typed-source.test.ts` — colocated tests.

**Modified files:**
- `src/gen-data-types.ts` — read curated routes from `src/data/routes.ts` (not `dist/data/routes.js`); add a step that emits `dist/data/routes.js` + `dist/data/routes.d.ts` via the new helper, with `GENERATED_CONTENT_PREAMBLE` prepended to both.
- `src/gen-data-types.test.ts` — update default-deps stubs and add coverage for the new emission step.
- `README.md` — update the "data variant" section to describe `src/data/routes.ts` as the source-of-truth.
- `CLAUDE.md` — same update so the architecture description matches reality.
- `tsconfig.json` — no changes expected (the existing `include: ["src/**/*"]` already picks up `src/data/routes.ts`); confirm during typecheck.

**Files deleted at the end:**
- `dist/data/routes.js` and `dist/data/routes.d.ts` are deleted in Task 7 once Task 6 confirms the new pipeline regenerates equivalent content.

## Important Conventions

- **Banner consistency:** Both emitted files (`dist/data/routes.js` and `dist/data/routes.d.ts`) must start with `GENERATED_CONTENT_PREAMBLE` from `src/gen/generator.ts`. The TS compiler doesn't emit this — the helper prepends it via the `writeFile` callback parameter on `program.emit()`.
- **Type import path:** `src/data/routes.ts` imports from `../gen/schema-types.js` (note the `.js` extension required by ESM module resolution, even though the source file is `.ts`).
- **Compiler options:** When emitting via `ts.createProgram`, mirror the project's `tsconfig.json` settings (`module: ESNext`, `target: ESNext`, `moduleResolution: Bundler`, `strict: true`, `esModuleInterop: true`) so emission matches the rest of the codebase. Set `rootDir: src/`, `outDir: dist/` so the file lands at `dist/data/routes.js`.
- **Test style:** Per repo memory, skip trivial formatting tests. Focus on behavior (e.g. "emits the expected named exports", "the output round-trips through `import()`"), not whitespace, indentation, or banner-string matching.
- **Frequent commits:** One commit per task, conventional commit prefixes (`feat:`, `refactor:`, `test:`, `docs:`, `chore:`).

Create `src/gen/emit-typed-source.ts`:

```ts
import ts from 'typescript'
import { dirname, join, relative } from 'node:path'
import { mkdirSync, writeFileSync } from 'node:fs'

export interface EmitTypedSourceOptions {
  sourcePath: string
  rootDir: string
  outDir: string
  banner?: string
}

export interface EmitTypedSourceResult {
  jsPath: string
  dtsPath: string
  diagnostics: ts.Diagnostic[]
}

export function emitTypedSource(options: EmitTypedSourceOptions): EmitTypedSourceResult {
  const { sourcePath, rootDir, outDir, banner = '' } = options

  const compilerOptions: ts.CompilerOptions = {
    declaration: true,
    module: ts.ModuleKind.ESNext,
    target: ts.ScriptTarget.ESNext,
    moduleResolution: ts.ModuleResolutionKind.Bundler,
    esModuleInterop: true,
    strict: true,
    skipLibCheck: true,
    resolveJsonModule: true,
    rootDir,
    outDir,
    noEmit: false,
  }

  const program = ts.createProgram([sourcePath], compilerOptions)

  const written: string[] = []
  const emitResult = program.emit(undefined, (fileName, data) => {
    mkdirSync(dirname(fileName), { recursive: true })
    writeFileSync(fileName, banner + data)
    written.push(fileName)
  })

  const diagnostics = [
    ...ts.getPreEmitDiagnostics(program),
    ...emitResult.diagnostics,
  ]

  const rel = relative(rootDir, sourcePath).replace(/\.ts$/, '')
  const jsPath = join(outDir, `${rel}.js`)
  const dtsPath = join(outDir, `${rel}.d.ts`)

  return { jsPath, dtsPath, diagnostics }
}
```

export async function main(deps: Partial<MainDeps> = {}) {
  const { routesPath, schemaPath, outPath, readFile, writeFile, importRoutes, emitTypedSource, log } = { ...defaultDeps, ...deps };

  const routesModule = await importRoutes(routesPath);
  const routesByResource: Record<string, Record<string, RouteDef>> = {};
  for (const [k, v] of Object.entries(routesModule)) {
    if (k !== "default") routesByResource[k] = v as Record<string, RouteDef>;
  }

  const schemas: Record<string, RouteSchema> = JSON.parse(readFile(schemaPath));
  const output = generateDataTypes(routesByResource, schemas);
  writeFile(outPath, output);

  const emitResult = emitTypedSource({
    sourcePath: routesPath,
    rootDir: SRC,
    outDir: DIST,
    banner: GENERATED_CONTENT_PREAMBLE,
  });
  if (emitResult.diagnostics.length > 0) {
    const summary = emitResult.diagnostics.map(d => typeof d.messageText === 'string' ? d.messageText : d.messageText.messageText).join('\n')
    throw new Error(`emitTypedSource returned ${emitResult.diagnostics.length} diagnostic(s):\n${summary}`)
  }

  const s = summarizeCoverage(routesByResource, schemas);
  log(`Wrote ${outPath}`);
  log(`Wrote ${emitResult.jsPath}`);
  log(`Wrote ${emitResult.dtsPath}`);
  log(`  Methods total:        ${s.total}`);
  log(`  With any schema:      ${s.withSchema} (${(100 * s.withSchema / s.total).toFixed(1)}%)`);
  log(`  With request schema:  ${s.withOpts}`);
  log(`  With response schema: ${s.withResult}`);
}
```

## The `data` variant (Shogun)

The `data` variant covers Heroku's data services control plane (Shogun). Unlike `3.sdk`, Shogun does not publish a hyperschema, so the resource grouping in `src/data/routes.ts` is curated by hand. The body of `dist/data/types.d.ts` — every `*Opts` and `*Result` interface, plus the `HerokuClient` method signatures — is generated from request/response payloads captured during Shogun's spec suite. The runtime route registry at `dist/data/routes.{js,d.ts}` is compiled from `src/data/routes.ts` by the same pipeline, so `dist/` remains 100% reproducible.

### Pipeline

1. **Capture payloads in Shogun.** From a Shogun checkout, with the spec DB running:
   ```sh
   bundle exec rspec spec/shogun/endpoints
   bundle exec rake api_schemas:build
   ```
   This writes `tmp/api_schemas.json` (schemas keyed by `"VERB /path"`).

2. **Generate types in this repo.** Point the generator at the artifact:
   ```sh
   SHOGUN_SCHEMA_PATH=/path/to/shogun/tmp/api_schemas.json npm run generate:data
   ```
   This reads `src/data/routes.ts` for the curated resource grouping, emits `dist/data/types.d.ts`, and emits `dist/data/routes.{js,d.ts}` from the same source. Methods with no schema coverage are typed as `Promise<unknown>` and annotated with a `// TODO: no spec coverage` comment.

### What the generator preserves

The grouping in `src/data/routes.ts` is the source of truth. The generator never invents new resources or moves methods between resources — it only fills in `Opts`/`Result` types from the schema artifact. To add or rename a resource, edit `src/data/routes.ts` and re-run the generator. **Do not edit `dist/data/routes.{js,d.ts}` directly** — those files are regenerated on every `npm run generate:data` invocation.
